\chapter{Introduction}\label{ch:intro}
\section{What is \SaulPOS?}
\SaulPOS{} is a point-of-sale system designed to process transactions, control inventory, manage customers, and generate operational reports. The primary goal is consistency: every sale, discount, and return should be traceable.

\section{Who is this book for?}
This manual is intended for:
\begin{itemize}
  \item Cashiers with basic computer skills.
  \item Supervisors who approve exceptions and close shifts.
  \item Administrators who configure catalog, taxes, and users.
  \item Owners who review performance and enforce operational rules.
\end{itemize}

\section{Key benefits}
\begin{itemize}
  \item Repeatable sales workflow.
  \item Clear discount, tax, and return rules.
  \item Better inventory control and lower shrinkage risk.
  \item Better decision-making through reliable reports.
\end{itemize}

\chapter{Installation}\label{ch:installation}
\section{Prerequisites}
\begin{itemize}
  \item Java 21
  \item Maven 3.9 or newer
  \item PostgreSQL running locally
\end{itemize}

\section{Clean local installation}
\begin{enumerate}
  \item Create local database and user:
\begin{verbatim}
psql -U postgres <<'SQL'
CREATE USER saulpos_dev WITH PASSWORD 'saulpos_dev';
CREATE DATABASE saulpos_dev OWNER saulpos_dev;
GRANT ALL PRIVILEGES ON DATABASE saulpos_dev TO saulpos_dev;
SQL
\end{verbatim}

  \item Start backend with the dev profile (this applies Flyway migrations automatically):
\begin{verbatim}
cd pos-server
SPRING_PROFILES_ACTIVE=dev mvn spring-boot:run
\end{verbatim}

  \item Validate server status and migration metadata:
\begin{verbatim}
curl -sSf http://localhost:8080/actuator/health
psql -U saulpos_dev -d saulpos_dev -c \
"SELECT version, description, success FROM flyway_schema_history ORDER BY installed_rank;"
\end{verbatim}

  \item Bootstrap first login and base operational data:
\begin{verbatim}
psql -U saulpos_dev -d saulpos_dev -f ops/scripts/bootstrap-local.sql
\end{verbatim}

  \item Start client:
\begin{verbatim}
cd pos-client
mvn javafx:run
\end{verbatim}
If your API is not on \texttt{\detokenize{http://localhost:8080}}, set:
\begin{verbatim}
mvn javafx:run -Dsaulpos.api.base-url=http://localhost:9090
\end{verbatim}
\end{enumerate}

\section{Initial bootstrap defaults}
After running \texttt{\detokenize{ops/scripts/bootstrap-local.sql}}:
\begin{itemize}
  \item Login user: \texttt{admin}
  \item Login password: \texttt{Pass123!}
  \item Demo merchant/store/terminal: \texttt{DEMO}, \texttt{DEMO-STORE}, \texttt{POS-01}
\end{itemize}

\section{What migrations seed by default}
\begin{itemize}
  \item Seeded by migration chain: core roles, permissions, and selected reason-code catalogs.
  \item Not seeded by migration chain: login users, merchant/store/terminal hierarchy, product catalog.
\end{itemize}

\section{How migration updates are done}
\begin{enumerate}
  \item Add a new file in \texttt{\detokenize{pos-server/src/main/resources/db/migration}} using the next version format: \texttt{\detokenize{V43__description.sql}}.
  \item Never edit already-applied migration files.
  \item Restart backend; Flyway applies only pending versions and records them in \texttt{flyway\_schema\_history}.
\end{enumerate}

\chapter{Core Concepts}\label{ch:concepts}
\section{Products and services}
A \TermProduct{} is a physical item and impacts stock. A \TermService{} usually does not change stock, but it can still affect taxes and reports.

\section{Categories and subcategories}
Categories support product discovery, promotion targeting, and reporting. Example: \emph{Beverages} $\rightarrow$ \emph{Juices}.

\section{Customers}
Customer records store purchase history and contact details. If enabled by your administrator, credit/receivable workflows may also be available.

\section{Price, cost, and margin}
\begin{longtable}{p{0.28\textwidth}p{0.66\textwidth}}

\midrule
\endhead

\end{longtable}

\section{Discounts}
\subsection{Percentage vs fixed amount}
\begin{longtable}{p{0.30\textwidth}p{0.30\textwidth}p{0.34\textwidth}}
\toprule
\textbf{Type} & \textbf{\ExampleWord} & \textbf{\ResultWord}\\
\midrule
\endhead

Fixed amount & \$5.00 on \$50.00 & New subtotal \$45.00.\\
\bottomrule

\subsection{Line-level vs ticket-level}
\begin{itemize}
  \item \textbf{Line-level}: applies to one item line.
  \item \textbf{Ticket-level}: applies to the full sale.
\end{itemize}

\subsection{Stacking, rounding, and approvals}
\begin{warningbox}
Default best practice: do not stack multiple manual discounts on the same line unless this policy is explicitly approved.
\end{warningbox}

\begin{tipbox}
Define one rounding policy and keep it stable. Switching rounding rules mid-period creates reconciliation issues.
\end{tipbox}

\SaulFigure{\ShotPath{concepts_discount_line_01.png}}{Screen: Applying a line-level discount}{fig:concepts-discount}

\section{Taxes}
\begin{itemize}
  \item \textbf{Tax-inclusive}: listed prices already include tax.
  \item \textbf{Tax-exclusive}: tax is added at checkout.
  \item \textbf{Exemptions}: available if enabled by your administrator and legally justified.
\end{itemize}

\section{Inventory}
Base concepts:
\begin{itemize}
  \item \textbf{Stock on hand}: currently available units.
  \item \textbf{Adjustment}: correction after discrepancy.
  \item \textbf{Receiving}: stock intake from purchase orders.
  \item \textbf{Transfer}: branch-to-branch movement, if enabled.
\end{itemize}

\section{Payment methods}
Common methods include cash, card, and bank transfer. If enabled by your administrator, split payment is available.

\section{Returns and refunds}
Return windows are configurable. Always record reason, operator, and original sale reference when available.

\section{Roles and permissions}
\begin{itemize}
  \item \TermCashier: creates and collects sales.
  \item \TermSupervisor: approves exceptions and validates closures.
  \item \TermAdmin: configures catalogs, taxes, users, and policies.
\end{itemize}

\chapter{Standard Operating Procedures (SOP)}\label{ch:sop}
\section{SOP 01: Sale with customer queue}
\begin{enumerate}
  \item Keep one active sale per cashier.
  \item Scan, confirm quantity, and announce subtotal before payment.
  \item If the customer requests changes, stop payment and re-confirm final amount.
  \item Deliver receipt before starting the next customer.
\end{enumerate}

\section{SOP 02: Price mismatch at checkout}
\begin{enumerate}
  \item Pause the transaction.
  \item Verify catalog price and shelf evidence.
  \item Apply active policy (displayed price vs system price per internal rule).
  \item Save an operational note for catalog follow-up.
\end{enumerate}

\section{SOP 03: Network or service outage contingency}
\SaulFigure{\ShotPath{sop_network_contingency_01.png}}{Screen: Network outage contingency workflow}{fig:sop-network}

\begin{enumerate}
  \item Determine whether the issue impacts one register or the full store.
  \item Notify supervisor and activate local protocol.
  \item If enabled by your administrator, continue in contingency mode.
  \item Reconcile pending transactions after service recovery.
\end{enumerate}

\begin{warningbox}
Never duplicate charges manually. Before retrying payment, verify final status of the original ticket.
\end{warningbox}

\section{SOP 04: Daily close with multiple registers}
\begin{enumerate}
  \item Run pre-close at each register.
  \item Consolidate sales, discounts, returns, and cash withdrawals.
  \item Document variances and responsible users.
  \item Close each register, then execute store-level close.
  \item Export key reports (sales, taxes, discounts, returns).
\end{enumerate}

\begin{tipbox}
A short 10-minute end-of-day review helps teams fix recurring issues before the next shift.
\end{tipbox}
